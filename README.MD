**Packages**
Package (пакет) — это логическая группировка классов, которая предотвращает конфликты имён и
упрощает навигацию.
Объявление package ru.mentee.power; в начале файла говорит компилятору: «класс принадлежит
пространству имён ru.mentee.power»,
и IDE автоматически размещает файл в папке src/main/java/ru/mentee/power/.
В Java имя класса формируется как «полное имя пакета + имя класса» (Fully Qualified Name, FQN):
ru.mentee.power.devtools.ProgressDemo.

Зачем это нужно: Без пакетов два класса с одинаковым именем (например, Progress из модуля A и
Progress из модуля B)
столкнутся при импорте, и компилятор не сможет различить их.
Пакет создаёт уникальную область видимости; правило обратного доменного имени (ru.mentee.power)
гарантирует глобальную уникальность, так как доменное имя зарегистрировано владельцем.

Ограничения: Структура папок должна точно повторять имя пакета: если в коде написано package
ru.mentee.power,
файл обязан лежать в src/main/java/ru/mentee/power/, иначе компилятор выдаст ошибку «package does
not match directory».
IntelliJ IDEA автоматически синхронизирует структуру при создании пакета через New → Package.

### Git локальный цикл

1. Все файлы проекта и изменения хранятся в Рабочей директории (Working Directory).
   Когда вы редактируете файл и сохраняете его, изменения сначала попадают сюда.

2. Те изменения из рабочей директории, которые хотите включить в следующий коммит необходимо
   поместить в Индекс (Staging Area).
   Это промежуточная область. Для перемещения изменений из рабочей директории в индекс используется
   команда git add.
   Изменения в рабочей директории, которые не были добавлены в индекс, не попадут в коммит.

3. Место на вашем компьютере, где хранится вся история ваших коммитов(снимков состояния проекта)
   называется Локальный репозиторий (Local Repository).
   Команда git commit создает новый коммит, который фиксирует все изменения, находящиеся в индексе,
   и добавляет его в историю локального репозитория.

## Кодстайл-гайд проекта devtools

Проект следует правилам Google Java Style Guide с адаптацией.
Автоматическая проверка: ./gradlew checkstyleMain

### 1. Именование методов: camelCase

До:    public void add_student(Student s) { }
После: public void addStudent(Student student) { }

Почему: Java Convention требует camelCase для методов.
Источник: https://google.github.io/styleguide/javaguide.html#s5.3-camel-case

### 2. Пробелы после if/for/while

До:    if(condition) {
После: if (condition) {

Почему: улучшает читаемость, отделяет ключевое слово от выражения.
Источник: Oracle Code Conventions — Whitespace

### 3. Длина строки: максимум 120 символов

До:    public List getStudentsFromSpecificCityWithVeryLongName...
После: public List getStudentsByCity(String city) {

Почему: длинные строки затрудняют чтение в редакторе и при code review.
Источник: https://google.github.io/styleguide/javaguide.html#s4.4-column-limit

### 4. Порядок импортов

До:    import java.util.List; import java.util.ArrayList; import java.io.File;
После: import java.io.File; import java.util.ArrayList; import java.util.List;

Почему: алфавитный порядок упрощает поиск импортов.
Источник: IntelliJ IDEA → Code → Optimize Imports

### 5. Фигурные скобки для if

До:    if (condition) doSomething();
После: if (condition) { doSomething(); }

Почему: скобки обязательны даже для однострочных блоков.
Источник: https://google.github.io/styleguide/javaguide.html#s4.1.1-braces-always-used

## Continuous Integration (CI)

Практика автоматической интеграции кода из разных веток в основную ветку (обычно master или main)
с обязательной проверкой качества на каждом шаге.

В контексте Java-проектов типичный CI-пайплайн включает несколько этапов: checkout кода
(получение исходников из репозитория), настройка окружения (установка Java, Gradle),
проверка стиля кода (Checkstyle), запуск юнит-тестов (./gradlew test), сборка проекта
(./gradlew build). Если любой из этапов падает, весь workflow считается неуспешным,
и PR помечается красным статусом. Ревьюер видит этот статус в интерфейсе PR и может
кликнуть на него, чтобы посмотреть логи и понять, что именно сломалось.

Триггеры (events) определяют, когда запускается workflow. Самые частые триггеры:

- on: push (при любом push в указанные ветки),
- on: pull_request (при открытии, обновлении или ре-открытии PR),
- on: schedule (по расписанию, например, каждую ночь).

Можно комбинировать триггеры и указывать фильтры по веткам.

Jobs и steps — это структурные единицы workflow. Job — это набор шагов (steps),
которые выполняются на одном runner'е. Шаги выполняются последовательно,
и если хотя бы один шаг падает, весь job считается неуспешным.
Внутри job можно использовать готовые действия (actions) или запускать shell-команды.
Например, типичный job для Java-проекта выглядит так:

1) actions/checkout@v4 (склонировать код),
2) actions/setup-java@v4 (установить Java 25 Temurin),
3) run: ./gradlew --no-daemon checkstyleMain test build (запустить проверки и сборку).

Флаг --no-daemon отключает Gradle Daemon, что экономит память и избавляет от проблем с зависшими
процессами в CI-окружении.

Статус-чеки в pull request — это визуальное представление результатов CI.
Когда workflow завершается, GitHub отображает статус в секции Checks внизу страницы PR.
Если все проверки прошли успешно, вы видите зелёные галочки и кнопку Merge становится активной
(если настроены branch protection rules).
Если хотя бы одна проверка упала, вы видите красный крестик
и можете кликнуть Details, чтобы посмотреть логи.

**AI-2 ([deepseek]):**
План реализации валидации email:

Шаг 1: Определить место для валидации
Добавьте валидацию в конструктор или компактный конструктор (если Student — record). Это гарантирует, что объект никогда не создастся с некорректным email. Валидация должна быть первой операцией в конструкторе.

Почему именно так: Принцип fail-fast (быстрое обнаружение ошибок) — невалидный объект не должен существовать в принципе. Если разнести валидацию и создание, можно получить объект в некорректном состоянии.

Шаг 2: Проверка на null и пустую строку
Сначала проверьте, что email не null и не пустой (или не состоит только из пробелов). Выбросьте IllegalArgumentException с понятным сообщением, если проверка не пройдена.

Почему именно так: null и пустая строка — самые очевидные нарушения. Проверять их нужно в первую очередь, так как последующие проверки (вызов методов у строки) могут выбросить NullPointerException — это сделает сообщение об ошибке менее понятным для вызывающего кода.

Шаг 3: Проверка наличия символа @
Проверьте, что строка содержит символ @ (используйте метод contains()). Если @ отсутствует — выбросите исключение.

Почему именно так: @ — обязательный разделитель между локальной частью и доменом. Это минимальное требование к email-адресу. Без него строка технически не может считаться email.

Шаг 4: Проверка наличия точки после @
Найдите позицию символа @ (метод indexOf('@')). Затем проверьте, что после этой позиции есть символ . (точка). Убедитесь, что точка находится не сразу после @ и не является последним символом.

Почему именно так: Домен должен содержать хотя бы одну точку (например, gmail.com, yandex.ru). Проверка позиции важна: адрес типа user@domain. (точка в конце) или user@.domain (точка сразу после @) — технически некорректны. Сейчас вы делаете простую проверку, но закладываете основу для будущего расширения.

Шаг 5: Выбросить исключение с информативным сообщением
При каждом нарушении выбрасывайте IllegalArgumentException с разными сообщениями:

"Email не может быть null или пустым"

"Email должен содержать символ @"

"Email должен содержать точку в доменной части (после @)"

Почему именно так: Разные сообщения помогут при отладке и написании тестов. Вы сможете точно определить, какое именно условие нарушено, а не гадать по одному общему сообщению "Некорректный email".

Дополнительные рекомендации:
Порядок проверок имеет значение:

null/пустая строка (самая грубая ошибка)

наличие @ (структурная ошибка)

наличие точки после @ (детальная ошибка)

Что не нужно делать сейчас:
Не проверяйте наличие символов до @ (например, "user@")

Не проверяйте, что точка не стоит сразу после @ (оставьте на следующий этап)

Не проверяйте допустимые символы (спецсимволы, кириллицу и т.д.)

Как протестировать (мысли вслух, не код):
Создайте тесты с корректными email (пример: "ivan@example.com")

Создайте тесты с null, пустой строкой, строкой из пробелов

Создайте тесты с email без @ (пример: "ivan.example.com")

Создайте тесты с email без точки после @ (пример: "ivan@examplecom")

Создайте тесты с точкой до @ (пример: "ivan.ivanov@example.com" — должно пройти валидацию, так как точка до @ допустима)

Ограничения текущего подхода (важно понимать):
Простая проверка пропустит очевидно невалидные адреса, например:

"user@.com" (точка сразу после @)

"user@domain." (точка в конце)

"user@domain..com" (две точки подряд)

Но для первого этапа этого достаточно. Вы всегда можете усложнить валидацию позже, добавив проверки для этих случаев.

Сообщение для невалидного массива
